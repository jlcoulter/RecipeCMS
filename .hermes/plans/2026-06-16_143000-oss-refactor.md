# StaticCMS: Open-Source General-Purpose Static Site Generator Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Refactor the RecipeCMS into a general-purpose, open-source static site generator where the content type is user-defined (recipes, cocktails, books, etc.), taxonomy fields are configurable, and styling is customizable via CSS overrides and config-level variables.

**Architecture:** A single Python script (`build.py`) reads a user config file (`site_config.json`) that defines the content type name, taxonomy fields, styling variables, and site metadata. The generator scans content JSON files, builds in-memory indexes, and emits static HTML + CSS + JS. The content JSON schema is driven by the config -- whatever taxonomy fields the user defines in config become first-class linked/faceted pages. A default CSS is always generated from the config's color variables, and users can drop a `custom.css` file to override anything.

**Tech Stack:** Python 3.11+, no external dependencies (stdlib only: json, os, re, math, random, copy, datetime), static HTML/CSS/JS output.

## Current State Analysis

The existing codebase has:
- `python/main.py` (951 lines) -- monolithic script with all generation logic
- `python/utils/` -- 9 modules that are all tightly coupled via `from .global_variables import *` (module-level mutable globals)
- Hardcoded strings: "Just My Cooking", "justmy.cooking", "Recipes", recipe-specific field names
- Hardcoded CSS inline in `html_header.py`, `html_footer.py`, `html_recipe_blocks.py`
- Hardcoded JS inline in `javascript.py` with site-specific URLs baked in
- Recipe-specific logic: `ingredient_map`, `cuisine_map`, `category_map`, `equipment_map`, `diet_map` all hardcoded as separate globals with duplicate code per map
- `environment_variables.json` (gitignored) with only 3 fields: `search_path`, `url`, `header`
- `recipe_template.json`, `listacle_template.json`, `shared_notes_template.json` at project root
- ~90 recipe JSONs in `data/` and `recipes/` directories
- `style.css` in `recipes/` with CSS custom properties already (good starting point)

## Key Design Decisions

### 1. Content Type is Config-Driven
- The word "recipe" is replaced everywhere by a configurable `item_name` (singular) and `item_name_plural` (plural)
- Example: `item_name: "cocktail"`, `item_name_plural: "cocktails"` produces "All Cocktails", "Cocktails by Ingredient", etc.
- The HTML `<title>` suffix comes from `site_name` config field

### 2. Taxonomy Fields are User-Definable
- The config defines an array of taxonomy fields, each with: `key`, `label`, `label_singular`, `link_prefix_text`, `link_suffix_text`
- Example config:
  ```json
  {
    "taxonomies": [
      {
        "key": "recipeCuisine",
        "label": "Cuisines",
        "label_singular": "Cuisine",
        "link_prefix_text": "Recipes that are",
        "link_suffix_text": "(roughly) in origin"
      },
      {
        "key": "diet",
        "label": "Dietary",
        "label_singular": "Diet",
        "link_prefix_text": "Recipes for a",
        "link_suffix_text": "diet"
      }
    ]
  }
  ```
- The generator dynamically builds maps and pages for each taxonomy -- no more `cuisine_map`, `category_map` etc. as separate hardcodes
- Special handling for `ingredientKeywords` linking within ingredient lists stays (it's the one field with intra-item text linking logic), but is itself configurable as a taxonomy entry with `type: "ingredient"` to flag the special behavior

### 3. Styling is Config + Override
- `site_config.json` contains a `style` object with CSS variable overrides
- The generator emits `style.css` using a default template that injects the config values into `:root` CSS variables
- If a `custom.css` file exists next to the output, the HTML includes it after `style.css` (cascade wins)
- The existing CSS custom properties (light/dark theming) are preserved and extended

### 4. No External Dependencies
- Remain pure stdlib Python. No Jinja2, no template engine.
- HTML is still constructed via Python f-strings but extracted into template functions in dedicated modules

class LinkFormatter:
    def __init__(self, config):
        self.config = config
    
    def format_link(self, link):
        if self.config.remove_html_extension:
            root_index_link = f"{self.config.site_url}/index.html"
            if link == root_index_link:
                return self.config.site_url
            if link.endswith('/index.html'):
                if link.split('/')[-1] == 'index.html':
                    return link[:-10]
            if link.endswith('.html'):
                return link[:-5]
        return link
    
    def get_keyword_link(self, keyword):
        keyword_slug = keyword.lower().replace(' ', '_')
        keyword_link = self.format_link(f"{self.config.site_url}/keywords/{keyword_slug}.html")
        return keyword_link
    
    def get_item_link(self, item_folder):
        item_link = self.format_link(f"{self.config.site_url}/{item_folder}/{item_folder}.html")
        return item_link
```

import json
import os

class ContentLoader:
    def __init__(self, config):
        self.config = config
    
    def load_items(self):
        """Walk content/items/ and load all item JSON files.
        Returns list of item dicts."""
        items = []
        for root, _, files in os.walk(self.config.items_path):
            for file in files:
                if file.endswith('.json'):
                    with open(os.path.join(root, file), 'r') as f:
                        items.append(json.load(f))
        return items
    
    def load_collections(self):
        """Walk content/collections/ and load all collection JSON files.
        Returns list of collection dicts."""
        collections = []
        for root, _, files in os.walk(self.config.collections_path):
            for file in files:
                if file.endswith('.json'):
                    with open(os.path.join(root, file), 'r') as f:
                        collections.append(json.load(f))
        return collections
    
    def load_shared_notes(self):
        """Walk content/shared_notes/ and load all shared notes JSON files.
        Returns dict keyed by folder name."""
        notes = {}
        for root, _, files in os.walk(self.config.shared_notes_path):
            for file in files:
                if file.endswith('.json'):
                    with open(os.path.join(root, file), 'r') as f:
                        data = json.load(f)
                        notes[data['folder']] = data
        return notes
```

**Objective:** Replace the hardcoded `cuisine_map`, `category_map`, `ingredient_map`, `equipment_map`, `diet_map` (and their `small_*` counterparts) with a single dynamic system driven by the config's `taxonomies` array.

```python
class TaxonomyIndex:
    """Builds and holds taxonomy maps dynamically from config.
    
    Each taxonomy has:
      - map: {value_string: [item_dict, ...]}  -- items indexed by taxonomy value
      - values: [value_string, ...]             -- flat list of all values (for search JS)
    """
    
    def __init__(self, config):
        self.config = config
        self.maps = {}    # key -> {value -> [items]}
        self.values = {}  # key -> [value_strings]
    
    def build(self, items):
        """Build all taxonomy maps from the loaded items."""
        for taxonomy in self.config.taxonomies:
            key = taxonomy['key']
            self.maps[key] = {}
            self.values[key] = []
        
        for item in items:
            for taxonomy in self.config.taxonomies:
                key = taxonomy['key']
                raw_value = item.get(key)
                if raw_value is None:
                    continue
                
                # Normalize to list
                values = [raw_value] if not isinstance(raw_value, list) else raw_value
                
                for value in values:
                    if value not in self.maps[key]:
                        self.maps[key][value] = []
                        self.values[key].append(value)
                    self.maps[key][value].append(item)
    
    def get_sorted_map(self, key):
        """Return map for a taxonomy key, sorted by item count descending."""
        if key not in self.maps:
            return {}
        return dict(sorted(self.maps[key].items(), key=lambda x: len(x[1]), reverse=True))
    
    def get_keyword_lastmod(self, key, value):
        """Get the most recent datePublished for items in a taxonomy value."""
        items = self.maps.get(key, {}).get(value, [])
        latest = "19900101"
        result = ""
        for item in items:
            date = item['datePublished'].replace("-", "")
            if int(date) > int(latest):
                latest = date
                result = item['datePublished']
        return result
    
    def get_all_keyword_links(self, formatter, limit=99999):
        """Generate keyword link HTML for all taxonomies (for sidebar / all-keywords page).
        Returns a dict of taxonomy_key -> list of (label, link_html) tuples."""
        result = {}
        for taxonomy in self.config.taxonomies:
            key = taxonomy['key']
            sorted_map = self.get_sorted_map(key)
            links = []
            count = 0
            for value, items in sorted_map.items():
                slug = value.lower().replace(' ', '_')
                link = formatter.format_link(f"{self.config.site_url}/keywords/{slug}.html")
                links.append((value, len(items), link))
                count += 1
                if count >= limit:
                    break
            result[key] = links
        return result
```

## Risks & Tradeoffs

1. **Ingredient linking complexity**: The plural-matching logic for ingredient keywords (main.py:259-293) is intricate and fragile. It needs to be preserved exactly as-is within the ingredient taxonomy handler. This is the one piece of domain-specific logic that stays even in the generic system -- it's flagged via `type: "ingredient"` in the taxonomy config.

2. **Backward compatibility**: The old `python/main.py` is deleted. Users who clone the repo will need to use the new `build.py`. This is intentional -- the old code is not a stable API.

3. **No template engine**: HTML is still built via Python f-strings. This is a deliberate choice to keep zero dependencies, but makes the template code harder to read than Jinja2. A future iteration could add Jinja2 as an optional dependency.

4. **Schema.org typing**: The JSON-LD schema type (currently "Recipe") is configurable but changing it to something like "Article" would require different schema fields. The generator currently only knows how to produce Recipe-schema structured data. For non-Recipe content types, the schema block may need to be simplified or made configurable per-field. This is noted as a future enhancement.

5. **Shared notes coupling**: The shared notes feature is still somewhat recipe-specific (it cross-links items that share notes). This generalizes fine as long as items share the same concept of "notes", but the naming is now generic.

## Open Questions

1. **Item schema flexibility**: Should the item JSON allow arbitrary additional fields beyond what the generator uses? Currently the generator only reads known fields. For full generality, we could add a `custom_fields` section to the config that maps extra JSON keys to display locations on the page. This is deferred to a future iteration.

2. **Collection image handling**: Collections currently reference an item folder name as their `image` field (e.g., `"image": "kedgeree"` means use `kedgeree/kedgeree_0.jpg`). This coupling works but is a bit odd. Should collections support their own standalone images instead? Deferred.

3. **Multi-language**: No i18n support is planned. All labels come from the config, so a user could manually set French labels, but there's no automatic translation. This is acceptable for v1.

4. **The `recipes/` directory after migration**: Should the old `recipes/` directory (which contains both source content and generated output mixed together) be fully deleted? It's currently the website root for deployment. After migration, `output/` becomes the deployable directory. This needs a clean break.
